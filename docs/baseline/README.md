# Оценка бейзлайнов и пайплайн валидации ассетов (Baseline Benchmarking & Evaluation)

Этот раздел содержит законченную систему автоматизированной генерации тестовых сэмплов различных baseline сценариев и A/B тестирования (Ablation Study) моделей генерации игровых 2D-ассетов для проекта **Auto-VideoGame-Assets-Pipeline**. 

---

## Работа с данными

Работа с EDA и обработке данных велась в полу-ручном режиме, процесс которого описан в 4 задании MFDP:

1.  **Исследовательский анализ данных (EDA):**
    *   **`eda_character_dataset_analysis.ipynb`** — ноутбук с глубоким статистическим анализом датасета персонажей. В нем исследуются распределения Booru-тегов, баланс классов, физическое разрешение картинок, удаление дубликатов и обоснование намеренного смещения данных (*Bias by Design* — уклон в characters).
    *   **`eda_props_dataset_analysis.ipynb`** — ноутбук с анализом датасета предметов и иконок заклинаний. Здесь исследуется распределение тегов-якорей, анализируются NLP-описания пространственных связей объектов и обосновывается выбор гибридной разметки tags + NLP.
2.  **Загрузка, очистка и подготовка данных:**
    *   В файле **`business_plan.md` (Раздел 2: «Сбор и подготовка данных»)** подробно описана архитектура подготовки:
        *   **Секция 2.1 (Синтетическое расширение):** как малый эталонный датасет из 50 ручных картинок персонажей был расширен до 2500+ вариантов с помощью библиотеки промптов в ComfyUI.
        *   **Секция 2.2 (Ручная модерация):** процедура *Human-in-the-loop* фильтрации, отбраковки мусора и анатомического брака, в результате которой датасет сократился до эталонных 1172 картинок.
        *   **Секция 2.3 (Разметка):** обоснование и реализация автоматической разметки Booru-style с помощью **WD-14 Tagger** для персонажей и гибридной разметки (теги + NLP от VLM LLaVA-7B) для предметов.

## 1. Архитектура baseline (Структура ноутбуков)

Пайплайн разбит на 7 специализированных Jupyter-ноутбуков для разделения зон ответственности и удобного проведения независимых тестов:

1.  **`nb_00_auto_generation.ipynb` — Движок генерации и LLM-расширение:**
    *   *Назначение:* Автоматизированное создание тестового набора данных (500 изображений — 5 сценариев по 100 штук).
    *   *Почему так сделано:* Ноутбук содержит REST/WebSocket клиент к локальному API ComfyUI. Он динамически считывает JSON-схемы воркфлоу, переопределяет положительные/отрицательные промпты, сид-политику (`idx * 7919`), разрешение (`512x512`), параметры KSampler (12 шагов, CFG 2.0, Euler Ancestral, Normal scheduler) и автоматически скачивает сгенерированные картинки из выходного буфера ComfyUI в папки проекта. 
    *   *Интеграция с LLM:* Реализует интеграцию с локальной Ollama (Mistral) для сценариев расширения промптов, а также содержит режим симуляции (`DRY_RUN = True`) для сквозного тестирования кода без GPU.
2.  **`nb_01_baseline1_benchmark.ipynb` — Baseline 1 (Pure Base):**
    *   *Тестируется:* Чистая базовая модель Anima Base (Nvidia Cosmos 2) + Turbo LoRA на простых промптах без стилевых адаптеров.
    *   *Зачем:* Служит нижней границей метрик (point of reference) — точкой отсчета для оценки прироста качества от LoRA.
3.  **`nb_02_baseline2_benchmark.ipynb` — Baseline 2 (Character LoRA - Simple):**
    *   *Тестируется:* Anima Base + Turbo LoRA + Character LoRA (`@sltn`) на простых промптах-тегах.
    *   *Зачем:* Проверяет, насколько эффективно стилевая LoRA, обученная по Data-centric подходу на персонажах с tag-разметкой, накладывает брендбук (цветные контуры Iro-toresu, Painterly Shading). 
    *   *Особенность:* Напрямую парсит бинарный заголовок `.safetensors` файла LoRA и автоматически логирует в MLflow параметры обучения (`lr`, `rank`, `alpha`, `epochs`, `optimizer`).
4.  **`nb_03_baseline3_benchmark.ipynb` — Baseline 3 (Props LoRA - Simple):**
    *   *Тестируется:* Anima Base + Turbo LoRA + Props LoRA (`@spll_icn`) на простых промптах предметов.
    *   *Зачем:* Оценивает стилизацию иконок заклинаний и магических кристаллов (сочное свечение, дымчатые текстуры на темном фоне). Считает KID относительно Props-референсов.
5.  **`nb_04_baseline4_benchmark.ipynb` — Baseline 4 (Character LoRA + LLM):**
    *   *Тестируется:* Character LoRA с промптами, расширенными с помощью LLM (Ollama/Mistral).
    *   *Зачем:* Проверяет гипотезу, улучшает ли добавление детальных NLP-описаний от LLM качество генерации персонажей.
6.  **`nb_05_baseline5_benchmark.ipynb` — Baseline 5 (Props LoRA + LLM):**
    *   *Тестируется:* Props LoRA с промптами, расширенными с помощью LLM.
    *   *Зачем:* Оценивает, насколько хорошо гибридный подход к обучению (разметка тегами + NLP LLaVA) у Props LoRA сочетается с NLP-описаниями от LLM при генерации эффектов.
7.  **`nb_06_final_comparison_benchmark.ipynb` — Финальный агрегатор результатов:**
    *   *Назначение:* Автоматически скачивает результаты всех выполненных Runs из локальной базы данных MLflow, строит сравнительный Bar Chart по KID, CLIP Score и LPIPS, выводит лучшие модели по каждой метрике и отправляет сводный Run в MLflow.

## 3. Ключевые выводы и инсайты (Data-centric анализ)

Анализ метрик выявил фундаментальные паттерны поведения генеративных моделей при различных стратегиях разметки и промптинга:

### А. Влияние LLM на генерацию персонажей (BL2 vs BL4) — LLM вредна ❌
*   **Результат:** При расширении промптов через LLM стилевой показатель KID **ухудшился на 32.6%** (вырос с `0.042` до `0.056`), а CLIP Score просел на `2.3%`.
*   **Почему так произошло:** Твоя Character LoRA обучалась строго на Booru-тегах (WD-14 Moat Tagger), и текстовый энкодер модели Anima Base привык связывать триггер `@sltn` с жесткой структурой тегов. Когда LLM превращает промпт в длинный текст на естественном языке, это "размывает" фокус текстового энкодера. Модель начинает отвлекаться на прорисовку фонов и побочных деталей из базы Cosmos 2, из-за чего фирменный стиль контуров и покраски деградирует.
*   **Рекомендация:** Для генерации персонажей **использовать только короткие Booru-теги и простые промпты (BL2)**. Использование LLM здесь не имеет практического смысла.

### Б. Влияние LLM на генерацию предметов/эффектов (BL3 vs BL5) — LLM полезна!  
*   **Результат:** LLM-расширение промптов предметов **улучшило стиль (KID) на 4.2%** (KID упал до `0.036` — лучший результат стилизации во всем проекте!). CLIP Score при этом снизился до `0.271`.
*   **Почему так произошло:** В отличие от персонажей, Props LoRA обучалась на **гибридной разметке** (Booru-теги + NLP описания пространственных связей от VLM LLaVA-7B). Текстовый энкодер Props LoRA умеет работать с естественным языком. Добавление богатых NLP описаний от LLM (свечение, градиенты, дымчатые текстуры) дало мощный синергетический эффект. Картинки стали невероятно сочными и живописными, идеально уложившись в ТЗ контражурного свечения.
*   **Рекомендация:** Для генерации предметов и спецэффектов **LLM-расширение дает наилучшую художественную выразительность (BL5)**, жертвуя лишь строгим соответствием CLIP Score из-за обилия абстрактных слов.

### В. Оценка качества обучения LoRA-моделей — Обучены блестяще! 🏆
*   **Стиль:** Обе LoRA-модели показали феноменальный результат: KID снизился на **42%** для персонажей и на **50%** для предметов по сравнению с базой. Это математически доказывает, что стилистика брендбука переносится железобетонно.
*   **Разнообразие (LPIPS):** Показатель LPIPS у обеих моделей держится на уровне `0.65 - 0.69` (критический порог коллапса равен `0.10`). Это означает, что **модели абсолютно не переобучены (Mode Collapse полностью отсутствует)**. Они сохраняют гибкость и богатство композиций базовой модели, меняя только стилистику прорисовки.
